# Анализ производительности

**Стенд:** AMD Ryzen 7 4800H, 16 логических ядер @ 2895 MHz, Windows 10 / WSL2 (Linux 6.6)  
**Тестовый файл:** `data/access.log` (1,080,890,899 байт ≈ 1.01 ГБ, 7,148,945 строк Apache Combined Log Format)  
**Компилятор:** GCC 15.2.0 (MSYS2), флаги Release (`-O2`)

> Linux-замеры будут добавлены после перезапуска `./scripts/run_benchmarks.sh` с исправленным бенчмарком
> (ветки `main` и `step1-sequential`). Фикс: `BM_ProcessFile` и `BM_ReadFile` → `->Iterations(1)->UseRealTime()`.

## Анализ по бенчмаркам

### BM_ParseLine — main: 4263 нс, step1: 414 нс

`std::regex` при каждом вызове выполняет полный проход NFA/DFA по строке ~200–400 символов: **4.3 µs/строку**.  
Ручной парсер на `string_view` + `std::from_chars` без аллокаций: **0.41 µs/строку** — ускорение **10.3×**.

Вклад в полный pipeline на 7.1М строк:
- main (regex): 4263 нс × 7.1М ≈ **30.3 сек** — почти весь BM_ProcessFile (40.9 сек)
- step1: 414 нс × 7.1М ≈ **2.9 сек** — только ~22% BM_ProcessFile (13.5 сек)

Оставшиеся 10.6 сек в step1 — это чтение файла в память (~3.3 сек) и накладные расходы агрегации + аллокация `vector<string>` для 1 ГБ данных.

### BM_ProcessFile — **40.9 сек** (main) vs **13.5 сек** (step1)

Сквозной pipeline на реальном 1 ГБ файле. Цель курса — **< 10 сек на 8 ядрах**.

- main: доминирует regex (>70% времени)
- step1: ускорение 3× относительно main, но узким местом стало чтение файла целиком в RAM
- Один поток из 16 доступных задействован

### BM_ReadFile — **3.40 сек** (main) / **3.33 сек** (step1)

Чтение 1 ГБ через `std::getline` в `vector<string>` занимает ~3.3–3.4 сек на Windows.  
Оба варианта идентичны — реализация reader одинакова.  
Текущий reader загружает весь файл в RAM (~1–2 ГБ). Переход на чанковое чтение устранит пиковое потребление памяти.

### BM_Accumulate — ~0.5 µs/запись

7.1М записей × 500 нс ≈ **3.6 сек** при последовательной агрегации.  
`unordered_map` по IP и URL — узкое место при многопоточности из-за конкуренции за запись; нужны thread-local копии с последующим merge.

## Что сдерживает производительность

| Причина | Влияние (step1) | Решение |
|---------|-----------------|---------|
| Весь файл в `vector<string>` | ~3.3 сек + пик памяти ~2 ГБ | Чанковое чтение |
| Однопоточность | 1 из 16 ядер | `std::thread` / thread pool |
| `unordered_map` без защиты | Не актуально сейчас, блокирует MT | Thread-local буферы + merge |
