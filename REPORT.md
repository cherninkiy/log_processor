# Анализ производительности — baseline (step1-sequential)

> Основан на release-замерах из `results/performance_report.md`.  
> Debug-результаты исключены — при отключённых оптимизациях данные не репрезентативны.

**Стенд:** AMD Ryzen 7 4800H, 16 логических ядер @ 2895 MHz  
**Тестовый файл:** `data/sample.log` (~10 МБ, ~71k строк Apache Combined Log Format)  
**Реализация:** `std::getline` + ручной парсер (`string_view` + `std::from_chars`), однопоточная

## Анализ по бенчмаркам

### BM_ParseLine — ~4 µs/строку (узкое место #1)

Причина: `std::regex` при каждом вызове выполняет полный проход NFA/DFA
по строке длиной ~200–400 символов. Это структурно дорого вне зависимости от данных.

Экстраполяция на 1 ГБ (~8 млн строк):
```
4 µs × 8 000 000 = 32 сек — только парсинг
```

**Ожидаемый эффект от замены на ручной парсер (`string_view` + `std::from_chars`):**
ускорение 10–20×, т.е. ~0.2–0.4 µs/строку → ~2–4 сек на парсинг 1 ГБ.

### BM_ProcessFile — 381 мс на 10 МБ

Сквозной pipeline (чтение + парсинг + агрегация) на 10 МБ.  
Экстраполяция на 1 ГБ: **~30–35 сек** — вдвое хуже цели 10 сек в однопоточном варианте.  
Доминирует парсинг (~90% времени).

### BM_ReadFile — 33 мс на 10 МБ

Экстраполяция на 1 ГБ: ~3.3 сек. Чтение само по себе не проблема.  
Текущая реализация загружает весь файл в `vector<string>` — при 1 ГБ это ~1–2 ГБ RAM.
Переход на чанковое чтение устранит пиковое потребление памяти.

### BM_Accumulate — ~200 нс/запись

Экстраполяция на 1 ГБ: 200 нс × 8М = ~1.6 сек. Значимо, но не доминирует.  
`unordered_map` по IP и URL будет узким при многопоточности из-за конкуренции за запись — нужны thread-local копии.

---

## Что сдерживает производительность

| Причина | Влияние | Решение |
|---------|---------|---------|
| `std::regex` в парсере | ~80–90% времени CPU | Ручной парсер: `string_view` + `from_chars` |
| Весь файл в памяти (`vector<string>`) | ~1–2 ГБ RAM на 1 ГБ лога | Чанковое чтение |
| Однопоточность | Используется 1 из 16 ядер | `std::thread` / thread pool |
| `unordered_map` без защиты | Не актуально пока, но блокирует переход к MT | Thread-local буферы + merge |

## Прогноз по шагам (1 ГБ, 8 ядер)

| Шаг | Изменение | Ожидаемое время |
|-----|-----------|----------------|
| step1 (regex) | baseline | ~30–35 сек |
| step1 (ручной парсер) | замена regex → ~15–20× быстрее парсинг | ~5–8 сек |
