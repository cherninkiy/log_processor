# Log Processor — многопоточный обработчик логов

**Log Processor** - это финальный проект курса **«Многопоточность»** из готовящегося образовательного блока **Expert C++**.  
Блок находится в стадии подготовки к пилотному запуску и пока не опубликован.

Проект представляет собой **эволюционную реализацию** высокопроизводительного обработчика логов Apache Combined Log. Каждая оптимизация (ручные потоки, атомики, пул, корутины) вынесена в отдельную ветку Git, что позволяет наглядно измерить вклад каждого улучшения.

Полная программа блока **Expert C++** охватывает управление памятью, сети, высокопроизводительные вычисления и другие темы — все материалы ориентированы на современный C++ и лучшие индустриальные практики.

## Задача

Разработать программу, обрабатывающую **1 ГБ логов** в формате Apache Combined Log **менее чем за 10 секунд** на 8‑ядерной системе.  
Программа должна собирать статистику:
- общее количество запросов;
- количество уникальных IP‑адресов;
- распределение HTTP‑кодов ответа;
- топ‑10 URL;
- объём переданных данных.

## Эволюционный подход

Проект реализуется поэтапно: каждая оптимизация вынесена в отдельную ветку Git, что позволяет измерить её вклад в производительность.  
Запланированные шаги: последовательная обработка, ручные потоки, атомики и thread‑local, пул потоков, параллельные алгоритмы STL, корутины C++20 для асинхронного ввода‑вывода. Финальная версия объединит лучшие практики и должна уложиться в целевые 10 секунд.

### Формат входных данных

Apache Combined Log Format — каждая строка имеет вид:

```
127.0.0.1 - frank [10/Oct/2000:13:55:36 -0700] "GET /apache_pb.gif HTTP/1.1" 200 2326 "http://referer.com/" "Mozilla/5.0 ..."
```

Поля: `ip`, `ident`, `user`, `time`, `method`, `url`, `protocol`, `status`, `bytes`, `referer`, `user_agent`.

## Требования к реализации

| Требование | Описание |
|---|---|
| **Корутины C++20** | Асинхронное чтение файла через корутину-генератор, не блокирующее потоки обработки |
| **Параллельный парсинг** | `std::execution::par` или пул потоков; файл делится на чанки по границам строк |
| **Отсутствие гонок данных** | Подтверждено ThreadSanitizer (TSAN); `std::atomic` для счётчиков, thread-local буферы для агрегации |
| **Отчёт о производительности** | Автоматически генерируемый отчёт с замерами до/после каждой оптимизации |
| **Качество кода** | Читаемый, документированный код, разбитый на логические модули |

### Критерии оценки

- Время выполнения < 10 секунд на 1 ГБ
- Эффективное использование всех ядер CPU (нет простоя потоков)
- Корректность: нет утечек памяти, нет гонок данных
- Читаемость и поддерживаемость кода
- Полнота тестов и бенчмарков

## Идея: эволюция проекта через ветки Git

Проект разрабатывается **поэтапно** — каждая ветка добавляет один новый механизм и позволяет измерить его вклад в производительность. Это превращает разработку в наглядное исследование.

| Ветка | Механизм | Ожидаемый результат |
|---|---|---|
| `step1-sequential` | Последовательное чтение, однопоточный парсинг | Baseline: ~60–90 сек на 1 ГБ |
| `step2-thread-manual` | `std::thread` + `std::mutex` для агрегации | Ускорение в ~4–6× на 8 ядрах |
| `step3-atomic-fixes` | `std::atomic` + thread-local буферы, устранение false sharing | +20–40% к шагу 2 |
| `step4-threadpool` | Пул потоков, очередь задач-чанков | Снижение накладных расходов, стабильная загрузка CPU |
| `step5-par-stl` | `std::for_each(std::execution::par, ...)` | Упрощение кода, сопоставимая производительность |
| `step6-coroutines-io` | Корутины C++20, ленивое чтение чанков | Совмещение I/O и обработки, демонстрация корутин |
| `main` | Оптимальная комбинация всех техник | Цель: < 10 сек |

### Шаг 1 — `step1-sequential`
Простейшая реализация: читаем файл строка за строкой (`std::getline`), парсим каждую строку регулярным выражением или вручную, обновляем счётчики. Никакой многопоточности. Служит **baseline** — отправной точкой для сравнения всех последующих шагов.

### Шаг 2 — `step2-thread-manual`
Файл делится на N примерно равных чанков (N = число ядер). Каждый чанк отдаётся отдельному `std::thread`. Главная задача: **правильно найти границу** — не разрезать строку пополам. Результаты каждого потока агрегируются через `std::mutex`. Проявляются первые проблемы: lock contention, false sharing.

### Шаг 3 — `step3-atomic-fixes`
Заменяем мьютексы там, где можно: простые счётчики → `std::atomic<uint64_t>`. Для сложных структур (`unordered_map` с URL/IP) — **thread-local копии** в каждом потоке, затем слияние в главном. `alignas(64)` устраняет false sharing между атомиками в смежных структурах.

### Шаг 4 — `step4-threadpool`
Вместо создания N потоков под каждый файл — **постоянный пул потоков**. Чанки ставятся в очередь задач (`std::queue` + condvar или lock-free очередь). Потоки берут задачи по мере готовности. Снижаются накладные расходы на `thread::join`, улучшается балансировка при неравных чанках.

### Шаг 5 — `step5-par-stl`
Заменяем ручное управление потоками на `std::for_each(std::execution::par, chunks.begin(), chunks.end(), process)`. Код становится компактнее. Требует Intel TBB как бэкенд (или встроенной реализации в MSVC). Сравниваем с шагом 4 по производительности.

### Шаг 6 — `step6-coroutines-io`
Вводим **корутину-генератор** `Generator<Chunk>`, которая ленивo читает файл чанками и `co_yield`-ит их в цикл обработки. Это позволяет совместить I/O и парсинг: пока один поток парсит предыдущий чанк, корутина уже читает следующий. Демонстрирует применение `co_yield`, `promise_type`, `suspend_always`.

### Финальная ветка — `final`
Объединяет лучшие практики: корутина для I/O + пул потоков для парсинга + атомики/thread-local для агрегации. Полная проверка TSAN, генерация отчёта.

## Сборка

```bash
git clone <url>
cd log_processor
mkdir build && cd build

# Release (для замеров)
cmake .. -DCMAKE_BUILD_TYPE=Release
cmake --build . -j

# Debug + ThreadSanitizer (для проверки корректности)
cmake .. -DCMAKE_BUILD_TYPE=Debug -DUSE_TSAN=ON
cmake --build .

# С тестами и бенчмарками
cmake .. -DCMAKE_BUILD_TYPE=Release -DBUILD_TESTS=ON -DBUILD_BENCHMARKS=ON
cmake --build . -j
```

## Запуск

```bash
# Обработка файла лога
./log_processor /path/to/access.log

# Генерация тестового файла (Python 3)
python scripts/generate_log.py --size-gb 1 --output data/test_1gb.log

# Запуск бенчмарков
./benchmarks/log_benchmark

# Запуск тестов
ctest --output-on-failure
```

## Производительность

Замеры проводятся на каждой ветке. Результаты сохраняются в `results/stepN.txt` и включают:
- Тестовую среду: CPU, RAM, ОС, версия компилятора, флаги оптимизации
- Время обработки при 1, 2, 4, 8 потоках
- Загрузку CPU (через `getrusage` или `perf`)
- Throughput (МБ/с)
- Подтверждение TSAN: нет гонок
